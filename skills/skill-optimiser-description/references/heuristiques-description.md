# Heuristiques de description

Une description solide devrait :

- partir de l usage, pas du theme ;
- mentionner l artefact ou le workflow precis ;
- inclure des verbes realistes que les utilisateurs vont sous-entendre ou employer ;
- mentionner des frontieres de perimetre importantes quand les faux positifs sont probables.

Exemple faible :

- "Skill pour Git et notes de version."

Meilleure forme :

- "Utiliser ce skill quand il faut generer des notes de version a partir de l historique de commits, de pull requests fusionnees ou de resumes de changements, surtout quand la sortie doit regrouper les changements par audience ou domaine de livraison."
